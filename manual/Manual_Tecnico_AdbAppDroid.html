AdbAppDroid v4.2

1. Visión General del Sistema

ADBAppKiller es una aplicación de escritorio para Windows que permite monitorear en tiempo real la aplicación activa en un dispositivo Android conectado por USB con Depuración USB activada. Aprovecha el protocolo ADB (Android Debug Bridge) para:

  • Identificar la app en primer plano con múltiples métodos redundantes.
  • Extraer metadatos del paquete: nombre, versión, permisos, rutas de APKs, etc.
  • Clasificar si es app de usuario o de sistema mediante heurísticas robustas.
  • Ejecutar acciones seguras: desinstalar, limpiar datos, extraer APKs, instalar y control remoto.

La app es 100% portable: no requiere instalación ni variables de entorno. Busca adb.exe y scrcpy.exe exclusivamente en:

platform-tools/adb.exe
scrcpy/scrcpy.exe

Si no existen, ofrece instalarlos localmente sin privilegios administrativos.

2. Arquitectura y Concurrencia

Hilos y Gestión de Eventos

La interfaz de usuario (Tkinter) se mantiene responsiva mediante el uso de hilos (threading.Thread). Las operaciones bloqueantes se ejecutan en segundo plano:

HiloFunciónIntervalo/Frecuencia
monitor_loop Consulta cada 1.5 segundos qué app está activa usando ADB. Cíclico (1.5s)
enhanced_device_monitoring Comprueba cada 3s si el dispositivo sigue conectado y autorizado. Cíclico (3s)
_full_initialization Carga inicial: paquetes del sistema/usuario, ADB, scrcpy. Una vez al inicio
acción específica Desinstalar, extraer APK, instalar, etc. Por demanda del usuario

Todas las actualizaciones de la UI se realizan mediante root.after(0, función) para evitar violar el hilo principal de Tkinter.

Caché Inteligente

Para reducir el uso de ADB y mejorar el rendimiento, se implementa un caché en memoria:

  • dumpsys_cache: almacena el resultado de adb shell dumpsys package [pkg] por 2 segundos.
  • packages_system y packages_user: listas de paquetes obtenidas una vez al inicio y actualizadas tras cambios (ej. desinstalación).

El caché se limpia automáticamente si supera 50 entradas, evitando fugas de memoria.

3. Detección de Aplicación Activa

El método get_foreground_app() intenta hasta tres estrategias para obtener el paquete activo:

  1. cmd activity top-activities
    → Extrae la primera actividad con regex: r"([a-zA-Z0-9._]+)/[a-zA-Z0-9._$]+".
  2. dumpsys activity activities
    → Busca "ResumedActivity" o "mResumedActivity".
  3. dumpsys window windows
    → Busca "mCurrentFocus" o "mFocusedApp".

Filtra paquetes inválidos, launchers y SystemUI. Solo retorna paquetes con formato válido según la regex:

re.fullmatch(r'[a-zA-Z][a-zA-Z0-9_]*(\.[a-zA-Z][a-zA-Z0-9_]*)*', pkg)

Si fallan los tres métodos, devuelve None y la UI muestra “Esperando detección…”.

4. Clasificación de Aplicaciones: Sistema vs Usuario

El método is_system_app_robust(pkg, apk_path, installer, uid, flags) aplica múltiples criterios:

CriterioDescripciónEjemplo
Ruta del APK Si comienza con /system/, /product/ o /system_ext/ → sistema. /system/app/Settings/Settings.apk
UID del proceso Si userId < 10000 → sistema (UIDs de usuario empiezan en 10000). UID 1001 = sistema
Flags de paquete Si contiene "SYSTEM" → sistema. flags=[SYSTEM, HAS_CODE, ...]
Instalador Si instalador es Play Store o PackageInstaller → usuario. com.android.vending
Base de datos local Compara contra listas cargadas al inicio: packages_system y packages_user. Obtenidas con pm list packages -s y -3

Este enfoque híbrido evita errores comunes en ROMs personalizadas (MIUI, OxygenOS, etc.).

5. Funcionalidades Principales – Desglose Técnico

5.1. Desinstalar Aplicación (uninstall_app)

  • Precondiciones: dispositivo conectado, app de usuario, confirmación del usuario.
  • Comando ADB: adb uninstall [package] (timeout: 20s).
  • Validación de éxito: verifica si la salida contiene "Success".
  • Post-proceso: actualiza la caché de paquetes y limpia el estado actual.
❌ Las apps de sistema no se pueden desinstalar (solo deshabilitar, pero eso no está implementado por seguridad).

5.2. Extraer APK (extract_apk)

  • Paso 1: obtiene rutas con adb shell pm path [pkg].
  • Paso 2: crea directorio temporal con tempfile.mkdtemp().
  • Paso 3: descarga cada APK con adb pull [ruta_remota] [ruta_local].
  • Paso 4:
    • Si hay 1 APK → guarda como .apk.
    • Si hay múltiples → empaqueta en ZIP y advierte al usuario que no es instalable directamente.

Soporta App Bundles (split APKs), comunes en apps grandes como YouTube o Gmail.

5.3. Instalar APK o Bundle (install_apk_bundle)

  • Selección: acepta .apk y .zip.
  • Si es ZIP: lo descomprime y busca archivos .apk internos.
  • Comando ADB: adb install-multiple [apk1] [apk2] ... (soporta bundles).
  • Timeout: 120 segundos.
  • Análisis de errores: método _parse_install_error() interpreta mensajes comunes.

Mensajes de error traducidos a lenguaje humano (ej. “Espacio insuficiente”, “Versión más antigua”).

5.4. Limpiar Datos (clear_app_data)

  • Comando: adb shell pm clear [package].
  • Efecto: borra /data/data/[pkg] y restablece la app como si fuera la primera ejecución.
  • Confirmación: requiere confirmación del usuario.

5.5. Exportar Informe (export_report)

Genera un archivo de texto estructurado con:

  • Información del dispositivo (marca, modelo, serie).
  • Metadatos de la app (nombre, paquete, versión, permisos, etc.).
  • Registro de eventos recientes (hasta 15 entradas).

Útil para auditorías de seguridad o diagnóstico técnico.

5.6. Lanzar scrcpy (launch_scrcpy)

  • Posicionamiento automático: abre la ventana justo a la derecha de la app.
  • Parámetros:
    --max-size=1024 --video-bit-rate=4M --max-fps=30
    --stay-awake --disable-screensaver
    --window-x=[posición] --window-y=[posición]
  • Proceso hijo: se guarda en scrcpy_process para poder terminarlo al cerrar.

6. Gestión de ADB y scrcpy

6.1. Detección Local

Los métodos get_adb_executable() y get_scrcpy_executable() solo buscan en el directorio de la app, garantizando portabilidad.

6.2. Instalación Automática

  • ADB: descarga platform-tools-latest-windows.zip desde Google y lo extrae en la raíz.
  • scrcpy: consulta la API de GitHub para obtener la última versión y descarga el ZIP correspondiente.

Ambos procesos se ejecutan en hilos separados y actualizan la UI en tiempo real.

🔒 Nunca modifica el sistema ni instala en C:\Windows o Program Files.

7. Seguridad y Robustez

  • Validación de paquetes: solo permite operar con nombres que cumplan con el estándar de Android.
  • Permisos peligrosos: se define un conjunto fijo (PELIGROSOS) y se resaltan visualmente.
  • Timeouts y reintentos: cada comando ADB tiene timeout (5-120s) y hasta 2 reintentos.
  • Codificación UTF-8: manejo explícito para evitar fallos en dispositivos con caracteres no ASCII.
  • Limpieza al salir: termina scrcpy y ejecuta adb kill-server.

8. Interfaz de Usuario (UI)

Diseño oscuro con contrastes altos. Elementos clave:

  • Etiqueta superior: muestra el paquete activo en verde brillante.
  • Panel de información: texto estructurado con colores por tipo de permiso.
  • Barra de estado: indica operaciones en curso (“Extrayendo APK…”, “Instalando…”).
  • Botones de acción: deshabilitados durante operaciones para evitar conflictos.
  • Mensajes de error: en rojo, con sugerencias (ej. “Acepta la clave RSA”).

9. Dependencias y Requisitos

9.1. Requisitos Mínimos

  • Windows 7+ (64 bits recomendado).
  • Python no necesario (aplicación empaquetada con PyInstaller).
  • Dispositivo Android con USB Debugging activado.
